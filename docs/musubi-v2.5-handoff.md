# Musubi v2.5 開発引き継ぎドキュメント

作成日: 2026-07-10(2026-07-12 個人環境に関する記述を削除)
用途: v2.5開発開始時の引き継ぎ。この1ファイルで開発計画の全体像を再開できることを目的とする。

---

## 1. プロジェクト概要

**Musubi** は、認知症当事者や外出に不安のある人の「行きたい」「参加したい」というニーズと、地域のサポーターの「手伝える」を、管理者の確認を通じて安全につなぐプロトタイプサービス。

- 最初の導入想定: 連携先ステークホルダーの地域カフェ周辺の地域活動支援
- 認知症カフェへの付き添いに限定せず、買い物・地域イベント・通院など外出支援全般に拡張できる設計とする
- 想定デモ相手: 地域側の受け入れ担当者(関係者への操作感・運用イメージ確認)

### 現状(v2)

- React stateのみで動作。DB・バックエンドなし(リロードでデータ消失)
- 既存プロジェクトの設定を流用していた → **v2.5では独立したプロジェクトとして作り直す**

### v2.5の位置づけ

「DBなしモック」ではなく、**Firestoreを使った実データ構造つきプロトタイプ(DBつきMVP)**。
画面リロード後も、登録済みニーズ・応募・マッチング状態を再取得できる。

---

## 2. v2.5 の目的(確認したいこと)

1. 利用者がニーズを登録できること
2. サポーターが既存ニーズに対して応募できること
3. 管理者が応募内容を確認してマッチングを成立させられること
4. マッチング成立後、利用者・サポーター側の画面に確定情報が反映されること
5. 実メール送信なしでも、本番運用時の通知イメージが伝わること

---

## 3. スコープ

### 実装するもの

- Firebase Hostingでの公開
- Firebase Authenticationによる簡易ログイン(**メールリンク認証を第一候補**。メールアドレス検証も兼ねられる)
- Cloud FirestoreによるDB保存
- 利用者によるニーズ登録
- サポーターによる既存ニーズへの応募
- 管理者によるマッチング成立(need.status→matched、offer.status→approved、matchingレコード作成)
- マッチング成立後の状態更新と、利用者・サポーター画面での確定情報表示
- 「本番ではメール送信される」旨の画面表示
- 利用者・サポーター向け注意事項の表示(確認ボタン付き)
- 全ログイン後画面の下部に緊急問い合わせ先表示
- サポーター向け情報の抽象化(目的地・詳細住所・氏名・連絡先はマッチング前非表示)

### 実装しないもの(次Phase送り)

- サポーターの空き時間登録(v2.5ではサポーターは必ず既存ニーズに応募する)
- needに紐づかないoffer
- 自動マッチング
- 実際のメール送信 / Cloud Functions / SMS / 電話連絡管理
- サポーターの一般公開登録・審査フロー
- 複数団体・複数地域対応
- 営業日カレンダー管理、キャンセル処理、当日チェックイン、チャット機能
- 目的地の表示要否・抽象化粒度の検討
- 登録者情報の正式項目化、電話番号対応
- 管理者によるサポーター招待リンク発行

### 重要な設計判断(確定事項)

1. **サポーターの空き時間登録は次Phaseへ送る**
2. **v2.5ではDB(Firestore)を入れる**
3. **サポーターには目的地を表示しない**(v2.5では目的地入力欄自体を削除し、支援カテゴリと補足だけ残す)
4. **公開情報と非公開情報はドキュメントレベルで分離する**(Firestoreはセキュリティルールで「同一ドキュメント内の一部フィールドだけ隠す」ことができないため)
5. 連絡手段はメール一択(電話番号は後Phase)

## 4. 基本フロー

1. 利用者(または代理入力者)がニーズを登録 → Firestoreに保存
2. サポーターが募集中ニーズを閲覧 → 対応できるニーズに応募 → Firestoreに保存
3. 管理者がニーズと応募を確認 → マッチング成立操作
4. need/offer/matchingの状態がFirestore上で更新される
5. 利用者・サポーターが画面を再表示すると確定情報が表示される
6. 画面上に「本番では双方にメールが送信されます」と表示(実送信はしない)

代理入力の許容: 画面操作者は本人とは限らない(家族、カフェ関係者、地域包括支援センター職員、ケアマネ等)。ただしv2.5では「登録者」情報は正式項目化しない(必要ならprivateNoteにメモ程度)。

## 5. 画面構成

### 5.1 トップ画面

Musubiの説明と3つの入口(利用者 / サポーター / 管理者)。

表示文言案:

> Musubi は、地域の中で「行きたい」「手伝える」を安全につなぐための試作サービスです。
> この画面は検討用プロトタイプです。
> v2.5ではデータ保存と状態更新を行いますが、実際のメール送信は行われません。
> 本番運用前に、入力項目・運用手順・安全確認の妥当性を確認するためのものです。

**必須の追加文言(DBを入れたことによる注意)**:

> このプロトタイプにはデータ保存機能があります。
> 本番運用前の試験では、実在する利用者の個人情報・詳細住所・緊急性のある情報は入力しないでください。

### 5.2 利用者画面

フロー: 注意事項確認 → 日時選択 → 支援カテゴリ選択 → エリア入力 → 詳細住所・名前・メール入力 → 登録完了

注意事項(入力開始前に表示、「確認しました」ボタンで進む):

> このサービスは、地域活動や外出に参加するための付き添い・見守りを調整するためのものです。
> サポーターに依頼できることは、外出中の付き添い・見守り・道案内・地域活動への参加補助です。
> 医療行為、身体介助、服薬管理、金銭管理、契約行為、緊急時の単独対応をサポーターに依頼することはできません。
> 困ったときは、画面下部の緊急問い合わせ先に連絡してください。

入力項目:

| 項目 | 備考 |
|---|---|
| 希望日 | 今日から14日先まで。営業日グレーアウトは必須ではない(将来対応) |
| 希望時間 | 開始希望・終了希望 |
| 支援カテゴリ | 地域活動への外出 / 買い物などの生活外出 / 通院などの外出 / その他 |
| 付き添いに関する補足 | 管理者確認用。サポーターへはマッチング前非表示 or 短く抽象化 |
| 大まかなエリア | 例: 西大井3丁目付近。サポーターに表示可 |
| 詳細住所 | 管理者のみ閲覧。マッチング成立後にのみサポーターへ表示 |
| 当人の名前 | 支援を受ける本人の名前 |
| 連絡先メールアドレス | v2.5ではメール一択 |

扱わない項目: 具体的な目的地(入力欄ごと削除)、電話番号、登録者種別、登録者連絡先、支援者・家族の詳細情報。

### 5.3 サポーター画面

フロー: 注意事項確認 → 募集中ニーズ一覧 → 公開情報確認 → 応募 → 応募完了 → マッチング成立後の状態確認

アクセス想定: 管理者から送られたサポーター用ログインリンク経由。メールリンク認証でログイン。

注意事項(初回表示、「確認しました」で進む):

> サポーターは、地域での外出や参加を支えるための付き添い・見守りを行います。
> 医療行為、身体介助、服薬管理、金銭管理、契約行為、緊急時の単独判断は行いません。
> 困った場合は、必ず管理者または画面下部の緊急問い合わせ先に連絡してください。
> マッチング成立前は、利用者の氏名、詳細住所、連絡先、具体的な目的地は表示されません。

**マッチング前に表示してよい情報(これだけ)**: 希望日 / 希望時間 / 大まかなエリア / 支援カテゴリ / 短い公開説明

表示例:

> 7月12日(日)13:00〜15:00
> 西大井3丁目付近
> 地域活動への外出
> 短時間の付き添いを希望しています

**マッチング前に表示しない情報**: 当人の氏名 / 詳細住所 / 連絡先メール / 具体的な目的地 / 詳細な事情 / 管理者向けメモ

応募操作: 各ニーズカードに大きな「このニーズに応募する」ボタン。応募時入力はサポーター名・メールアドレス・任意メッセージ。

応募後表示:

> 応募を受け付けました(試作)
> 管理者が内容を確認し、マッチングを承認します。
> 本番では、マッチング成立後にメールで通知されます。

### 5.4 管理者画面

フロー: ニーズ一覧 → 応募一覧 → 詳細情報確認 → マッチング成立 → 通知想定表示(「選ぶ → 確認する → 成立させる」に画面を分ける)

管理者が確認できる情報: ニーズ一覧 / 応募一覧 / 当人の名前 / 連絡先メール / 詳細住所 / 支援カテゴリ / 補足 / サポーター名 / サポーターメール / 応募メッセージ / 現在のステータス

マッチング成立操作:
1. ニーズを選択 → 2. 応募者を選択 → 3. 詳細確認 → 4. 「マッチング成立」ボタン
→ need.status を `matched` に、offer.status を `approved` に更新し、matchingレコードを作成

成立後の管理者画面表示:

> マッチングが成立しました。
> 本番では、利用者とサポーターの双方にメールが送信されます。
> この試作ではメールは送信されません。
> 利用者・サポーターは、画面を再読み込みすると確定情報を確認できます。

### 5.5 マッチング前後の状態表示

| 状態 | 利用者画面 | サポーター画面 |
|---|---|---|
| 成立前 | ニーズを受け付けました。現在、サポーターからの応募または管理者確認を待っています。 | 応募を受け付けました。管理者が内容を確認しています。 |
| 成立後 | マッチングが成立しました。本番では、登録されたメールアドレス宛に詳細が送信されます。この試作では、画面上で成立状態のみ表示しています。 | マッチングが成立しました。本番では、登録されたメールアドレス宛に詳細が送信されます。この試作では、画面上で確定情報を確認できます。 |

マッチング成立後にサポーターへ表示してよい情報: 日時 / 大まかなエリア / **詳細住所** / 管理者が必要と判断した補足 / 緊急問い合わせ先。**目的地は成立後も表示しない**。

### 5.6 緊急問い合わせ先(全ログイン後画面の下部)

> 困ったとき・当日のトラブル時:
> 結び運営担当(管理者)
> メール: example@example.com
> 電話: 本番運用時に設定

v2.5ではダミーでよい。本番前に実連絡先を設定する。

## 6. DB構成(Firestore)

公開/非公開をドキュメントレベルで分離した4コレクション。

### needsPublic/{needId} — サポーターにも見せてよい情報

```
id, status, date, startTime, endTime,
publicArea, supportCategory, publicSummary,
createdAt, updatedAt
```

例: `status: open, date: 2026-07-12, startTime: 13:00, endTime: 15:00, publicArea: 西大井3丁目付近, supportCategory: 地域活動への外出, publicSummary: 短時間の付き添いを希望しています`

**入れてはいけないもの**: 氏名 / メールアドレス / 詳細住所 / 目的地 / 詳細な事情

### needsPrivate/{needId} — 管理者のみ

```
needId, personName, personEmail,
privateAddress, privateNote,
createdByUid, createdAt, updatedAt
```

### offers/{offerId} — サポーターの応募

```
id, needId(必須), status,
supporterName, supporterEmail, message,
createdByUid, createdAt, updatedAt
```

status: `submitted` / `approved` / `rejected`

### matchings/{matchingId} — 成立したマッチング

```
id, needId, offerId, status,
matchedByUid, matchedAt, notificationStatus,
createdAt, updatedAt
```

status: `matched` / `notification_mocked` / `cancelled`
notificationStatus は実メール送信ではなく画面表示上の通知想定を表す。

## 7. 認証・権限

前提: 簡易でもよいのでログイン前提。Firestoreを誰でも読み書きできる状態には絶対にしない。

| ロール | 認証 | できること |
|---|---|---|
| 利用者 | メールリンク認証 | 自分が作成したneedの状態確認、成立後の自分のマッチング情報確認 |
| サポーター | 管理者から送られたリンク経由 + メールリンク認証 | 募集中ニーズの公開情報閲覧、自分のofferの状態確認、成立後の関係詳細情報のみ確認 |
| 管理者 | 管理者用ログイン | 全コレクション閲覧、マッチング成立操作 |

v2.5の管理者判定は簡易でよい: Firebase Authのカスタムクレーム、またはセキュリティルールに管理者UIDを直書きする方式で十分。

## 9. 技術・コスト方針(2026年7月時点の調査結果)

### Firebase(Sparkプラン = 0円)

- Firestore: 1GB、1日5万回読み取り・2万回書き込みまで無料 → デモ用途で余裕
- Hosting: 10GB/月ストレージ、1日360MB転送まで無料。`プロジェクト名.web.app` のURLが無料で付く
- Authentication: メールリンク認証は無料(メール送信もFirebase側が行う)
- Cloud FunctionsはBlaze必須だが、v2.5は実メール送信をしない設計なので不要 → **カード登録不要のSparkで完結**
- リージョンは `asia-northeast1`(東京)を選ぶ
- 将来Blazeに上げてもSparkの無料枠は維持される(次Phaseのメール送信対応でも実質数円〜数百円/月)

### GitHub(オープンソース公開の方針)

**公開して安全**。ただし以下を守る。

- Firebaseのweb設定(apiKey等)はコミットしてOK — 秘密鍵ではなく公開前提の識別子。安全性はセキュリティルール+Authで担保する
- **絶対にコミットしないもの**: サービスアカウントJSON(Admin SDK鍵)、`.env`内の管理者用シークレット → `.gitignore`に最初から記載
- セキュリティルールは公開されても問題ない(「ルールが見えても破れない」が正しい状態)。needsPrivateを管理者のみ読める、というルールが正しく書けているかがv2.5の生命線
- 個人情報入りのシードデータ・スクリーンショットをリポジトリに入れない

### 開発フロー推奨

- **Firebase Emulator Suite** をローカルで使う: 無料枠を消費せず、Firestore/Auth/Hostingをローカルで丸ごと再現。AIエージェントにテストさせるときも本番データを汚さない
- GitHub Actions(Firebase公式アクション)で mainマージ → 自動デプロイ(無料)

## 11. セキュリティルール要件(実装時の必須チェック)

- 未ログインユーザーは全コレクション読み書き不可
- `needsPublic`: ログイン済みユーザーは読み取り可。書き込みは作成者(createdByUid)と管理者のみ
- `needsPrivate`: 管理者のみ読み書き可(+作成時は作成者本人の書き込みを許可)
- `offers`: 作成は本人、読み取りは本人と管理者、status変更は管理者のみ
- `matchings`: 作成・更新は管理者のみ。読み取りは当事者(該当needの作成者、該当offerのサポーター)と管理者
- マッチング成立後のサポーターへの詳細住所表示は、matchedなmatchingの存在を条件に、専用の読み取りパス(またはマッチング成立時に管理者がサポーター向け確定情報ドキュメントを作る方式)で実現する — 「offerがapprovedならneedsPrivate全体を読ませる」のような雑な開け方をしない
- デプロイ前に `firebase emulators:exec` 等でルールのテストを書く

## 12. デモシナリオ(関係者向け)

1. トップ画面でMusubiの目的を説明
2. 利用者画面: 注意事項確認 → 14日以内の日程選択 → 支援カテゴリ選択 → エリア入力 → 詳細住所・名前・メール入力 → ニーズ登録
3. サポーター画面: 注意事項確認 → 募集中ニーズ閲覧 → **目的地・詳細住所・氏名・連絡先が見えないことを確認** → 応募
4. 管理者画面: ニーズ選択 → 応募者選択 → **管理者だけが詳細住所や連絡先を見られることを確認** → マッチング成立 → 本番ではメール送信される旨を確認
5. 利用者・サポーター画面を再読み込み → マッチング成立状態が表示されることを確認
